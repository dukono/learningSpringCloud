# 10.10 Compatibilidad, versionado y troubleshooting de contratos

---

Un contrato roto en producción interrumpe la comunicación entre servicios sin que haya habido ningún error de compilación en ninguno de los dos. El CDC detecta estas roturas en CI, pero solo si el pipeline está correctamente configurado y los contratos evolucionan con una estrategia deliberada de compatibilidad. La mayor fuente de fallos en proyectos con Spring Cloud Contract no es la herramienta, sino la falta de disciplina en tres áreas: versionar los contratos cuando la API cambia, diagnosticar correctamente por qué un stub no matchea, y resolver conflictos de entorno en CI. Este fichero cubre las roturas más frecuentes, su diagnóstico y las estrategias de rollback.

> [PREREQUISITO] Requiere haber completado al menos un ciclo CDC funcional con contracts en Git, plugin Maven/Gradle configurado y StubRunner en el consumer. Sin esa base, este fichero solo tiene valor conceptual.

## Mapa de errores frecuentes y su diagnóstico

Los errores de Spring Cloud Contract siguen patrones predecibles. La mayoría son detectables por el mensaje de error y el momento en que ocurren: durante la generación de tests (producer), durante la ejecución de StubRunner (consumer), o durante el startup de WireMock.

| Síntoma | Fase | Causa más probable | Solución |
|---|---|---|---|
| `Test generated by Spring Cloud Contract does not compile` | Producer CI | Cambio en el tipo de retorno sin actualizar el contrato | Actualizar `baseClass` o el contrato para reflejar el nuevo tipo |
| `Request was not matched` en StubRunner | Consumer CI | Contrato desactualizado o campo añadido sin `optional()` | Añadir `optional()` en contratos con campos nuevos; o actualizar el stub JAR |
| `Unable to find stub for [groupId:artifactId:version]` | Consumer CI | Stub JAR no publicado o `repositoryRoot` mal configurado | Verificar que el producer publicó el JAR antes del job del consumer; revisar `spring.cloud.contract.stub-runner.repository-root` |
| `Address already in use: 8080` en WireMock | Consumer CI | Dos StubRunners en el mismo JVM con el mismo puerto | Usar `spring.cloud.contract.stub-runner.ids` con puertos distintos: `groupId:artifactId:version:+8090` |
| `Contract validation failed: response body does not match` | Producer CI | Campo renombrado o eliminado del response | Versionar el contrato (v1 se mantiene, v2 introduce el cambio) |
| `Stub JAR contains no mappings` | Consumer CI | El producer nunca ejecutó el plugin Maven/Gradle correctamente | Verificar que el goal `generateTests` se ejecutó y que el JAR contiene `mappings/` |

## Ejemplo central: estrategias de compatibilidad al evolucionar contratos

El escenario más frecuente es añadir un campo nuevo al response del producer. Existen dos estrategias que mantienen la compatibilidad: `optional()` para campos que el consumer puede ignorar, y versionado explícito para cambios que rompen el contrato.

### Dependencias Maven

```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.cloud</groupId>
            <artifactId>spring-cloud-dependencies</artifactId>
            <version>2025.1.1</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

### Estrategia 1: campo opcional con `optional()` (backward compatible)

Cuando se añade un campo nuevo al response, el consumer existente no espera ese campo. Marcarlo como `optional()` permite que el stub genere mappings que aceptan requests con o sin ese campo, manteniendo los stubs existentes válidos.

```groovy
// contracts/v1/get-order.groovy — campo "discount" es nuevo y opcional
Contract.make {
    request {
        method GET()
        url '/orders/1'
    }
    response {
        status OK()
        body([
            id: 1,
            total: 100.0,
            // discount es un campo nuevo que consumers anteriores no esperan
            discount: $(consumer(optional(regex('[0-9]+\\.?[0-9]*'))),
                       producer(value(10.0)))
        ])
        headers {
            contentType(applicationJson())
        }
    }
}
```

> [CONCEPTO] `optional()` en el response de un contrato genera un mapping WireMock que acepta que el campo esté ausente en la petición del consumer. En el producer, el campo siempre se genera — `optional()` solo afecta a la validación del consumer.

### Estrategia 2: versionado de contratos (breaking change)

Cuando el cambio es incompatible (renombrar un campo, cambiar su tipo), la estrategia correcta es mantener el contrato v1 y añadir un contrato v2. El consumer migra a v2 en su propio sprint, mientras v1 sigue siendo válido.

```
contracts/
  v1/
    get-order.groovy     ← contrato original, v1 del stub JAR
  v2/
    get-order.groovy     ← contrato con el campo renombrado
```

```groovy
// contracts/v2/get-order.groovy — "total" renombrado a "amount"
Contract.make {
    request {
        method GET()
        url '/orders/1'
    }
    response {
        status OK()
        body([
            id: 1,
            amount: 100.0  // antes era "total" — breaking change
        ])
        headers {
            contentType(applicationJson())
        }
    }
}
```

```yaml
# application.yml del consumer migrando a v2
spring:
  cloud:
    contract:
      stub-runner:
        ids:
          # Usar versión "+" para coger el stub más reciente del producer
          - com.example:order-service:+:stubs:8090
        # Apuntar al stub v2; v1 sigue disponible en el mismo JAR como mapping separado
        stubs-mode: LOCAL
```

### Diagnóstico de `Request was not matched` con WireMock logs

Cuando StubRunner dice que la petición no matchea ningún stub, el primer paso es activar el log detallado de WireMock para ver qué diffieren.

```java
// Test con WireMock log activado
@SpringBootTest
@AutoConfigureStubRunner(
    ids = "com.example:order-service:+:stubs:8090",
    stubsMode = StubRunnerProperties.StubsMode.LOCAL
)
class OrderServiceConsumerTest {

    @Test
    void diagnoseNonMatchingRequest() {
        // Activar logs de WireMock para ver qué falló
        // En application-test.yml: logging.level.com.github.tomakehurst.wiremock=DEBUG
        // El log mostrará exactamente qué campo no coincide
    }
}
```

```yaml
# application-test.yml — activar WireMock debug
logging:
  level:
    com.github.tomakehurst.wiremock: DEBUG
```

> [ADVERTENCIA] El log de WireMock muestra el diff entre la petición recibida y los matchers del mapping. El campo `"distance"` en el log indica cuántos criterios fallan: un `distance=0` indica match perfecto; cualquier valor mayor indica qué campos no coinciden exactamente.

## Tabla de elementos clave

Los elementos que un desarrollador senior necesita para diagnosticar y prevenir roturas de contratos:

| Elemento | Descripción |
|---|---|
| `optional()` | Marca un campo como opcional en el matching del consumer; no afecta al producer |
| `$(consumer(...), producer(...))` | Define valores distintos para el stub (consumer) y el test generado (producer) |
| `stub-runner.ids: groupId:artifactId:version:classifier:port` | Formato completo del ID de stub; `+` como version = última disponible |
| `stub-runner.stubs-mode=LOCAL` | Carga stubs desde Maven local (`~/.m2`); requiere que el producer haya instalado el JAR con `mvn install` |
| `stub-runner.stubs-mode=REMOTE` | Descarga stubs desde el repositorio de artefactos; requiere `repository-root` configurado |
| WireMock `distance` en logs | Métrica de distancia entre petición real y matcher; 0 = match perfecto |
| `basePackageWithBaseClasses` | Estrategia del plugin para encontrar la clase base por paquete del contrato |

## Buenas y malas prácticas

**Hacer:**
- Versionar los contratos con `v1/`, `v2/` en directorios separados cuando hay breaking changes; permite que producer y consumer migren de forma independiente en sprints distintos.
- Activar `logging.level.com.github.tomakehurst.wiremock=DEBUG` en el consumer cuando un stub no matchea; el log muestra exactamente qué campo difiere del mapping.
- Publicar el stub JAR del producer en el pipeline CI antes de ejecutar los tests del consumer; si el consumer corre antes, fallará con "unable to find stub".
- Usar puertos distintos para cada StubRunner en el mismo JVM (`:+8090`, `:+8091`); el puerto `+N` indica "usar N si libre, o buscar el siguiente disponible".

**Evitar:**
- Eliminar contratos sin coordinar con los consumers: aunque el producer compile correctamente, los consumers que dependen de ese stub fallarán en CI hasta que actualicen su dependencia.
- Usar `version="+"` (latest) en producción sin un proceso de aprobación: un producer que publica un contrato roto bloquea automáticamente todos sus consumers en CI.
- Ignorar el `distance` en los logs de WireMock: un `distance=1` suele indicar un campo con formato ligeramente distinto (ej: `"2024-01-15"` vs `"2024-01-15T00:00:00Z"`) que requiere ajustar el matcher o el producer.
